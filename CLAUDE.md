## セッション開始時のルール

セッション開始時に必ず以下を実行すること：

```bash
git pull origin main
python3 scripts/fetch_definitions.py
```

その後、GitHubのissueを確認して作業の文脈を把握すること。

## セッション終了・区切り時のルール

作業の区切りや話題が変わるタイミングで、GitHubのissueに以下を記録すること：

- 完了したこと
- 未解決・次にやること
- 把握した現状の変化

## コミットメッセージのルール

コミットメッセージは必ず以下の形式で書くこと：

```
【プロジェクト】何を作っているかの説明（1行で簡潔に）
【課題】何の問題を解決したか
【解決】どう解決したか
【指示】ユーザーからのプロンプト（抽象的な言葉）→Claudeが具体化した内容
【手作業】ユーザーが自分で行った手動作業（なければ「なし」）
```

### 具体例

```
【プロジェクト】GitHubのコードをpushすると毎日20時にXへ自動投稿されるシステム【課題】bashのgrepでUnicodeエスケープが機能せずデータが入らない【解決】Python3でフィールド抽出する方式に変更【指示】「データが入らない」→post_to_notion.ymlのbash grep抽出をPython3のreモジュールに書き直せ【手作業】なし
```

### ルール

- `【】`の中の改行は不可。各項目は1行で書くこと
- 5つのタグがすべて存在する場合のみNotionに自動蓄積される（手作業は「なし」でも可）
- 【指示】は「ユーザーの言葉（抽象）→Claudeが具体化した内容」の形式で書くこと
- この形式に従うことで、GitHub ActionsがNotionに自動蓄積し、毎日20:00 JSTにXへ自動投稿される
- 定義を追加してpushした直後は `python3 scripts/fetch_definitions.py` を実行して最新の定義を取得すること
- PRをマージする際は必ず**rebaseマージ**を使うこと。squashマージはコミットメッセージが失われNotionに蓄積されないため禁止
- `.claude/settings.json` で権限制御（allow/deny）を設定済み

### 【プロジェクト】名の命名ルール

- セッション開始時にIssueを確認し、**現在進行中のプロジェクト名**を把握すること
- プロジェクト名はセッションをまたいで**必ず統一**すること
- 新しいプロジェクトを開始する場合のみ新しいプロジェクト名をつける
- プロジェクト名はIssueの作業メモに「現在のプロジェクト名：〇〇」として明記しておくこと

## ワークフロー失敗時のNotionリセットルール

ワークフローが失敗してNotionのステータスが中途半端な状態になっている可能性がある場合、会話の文脈から対象のプロジェクト・ステータスを特定し「〇〇のステータスをXXに戻しますか？」と確認してからfix_notion_data.ymlで対応する。全体リセットではなく特定のものだけを対象とすること。

## Notionデータ修正ルール

NotionのデータをClaude経由で変更・追加・修正する場合、まず`fix_notion_data.yml`での対応を提案する。

### 運用ルール
- ユーザーは何もしない。**ClaudeがGitHub MCP経由でfix_notion_data.ymlの変数セクションを書き換えてmainにpushすると自動実行される**
- 変数セクションは `fix_notion_data.yml` の `env:` ブロック内にある
- 同じ内容を再pushしてもトリガーされない場合は、コメント行（`# 最終更新:` など）を変えて差分を作ること
- 修正が終わったら変数セクションを次回用にリセットしておく（必須ではないが推奨）

### 変数の意味
- `MODE`: `update`（既存修正）または `create`（新規作成）
- `DATABASE`: `commit_log`（コミットログDB）または `project`（プロジェクト管理DB）
- updateモード: `FILTER_PROPERTY/VALUE`で対象を絞り込み、`UPDATE_PROPERTY/VALUE`で値を更新
- createモード: `CREATE_PROJECT_NAME`と`CREATE_STATUS`で新規プロジェクトを作成
- プロジェクトのステータスを「完了」にする際は必ず `UPDATE_PROPERTY_2: 完了日` / `UPDATE_VALUE_2: TODAY` を同時にセットすること

## ワークフロー構成

| ファイル | トリガー | 役割 |
|---|---|---|
| `post_to_notion.yml` | mainへのpush | コミットメッセージを解析してNotionに登録 |
| `post_to_x.yml` | 毎日20:00 JST | NotionのReadyアイテムをXに投稿 |
| `fix_notion_data.yml` | mainへのpush（ファイル変更時）| Notionデータの修正・新規作成 |
| `sync_to_repos.yml` | mainへのpush（CLAUDE.mdまたはpost_to_notion.yml変更時） | presetryとcolor-memoへ自動同期 |
| `sync_definitions.yml` | mainへのpush（CLAUDE.md変更時） | 定義一覧をNotionの定義DBへ自動同期 |

## 連携リポジトリ

- `pic-kn/presetry`
- `pic-kn/color-memo`
- `pic-kn/adhd-manager`

上記リポジトリはこのリポジトリと同じNotionデータベースを共有しており、`sync_to_repos.yml` によってCLAUDE.mdとpost_to_notion.ymlが自動的に最新版に保たれる。

## 会話から発火するルール

ユーザーの言葉からClaudeが状況を読み取り、自動的に動作を切り替える。

| ユーザーの言葉の例 | Claudeの動作 |
|---|---|
| 「なんか重い」「遅くなってきた」「リフレッシュして」 | `/compact` を実行して会話を圧縮する |
| 「定義」という言葉が会話に出た時 | `python3 scripts/fetch_definitions.py` を実行して最新の定義を取得する |
| 「ちょっと聞くだけ」「作業止めなくていい」「余談だけど」 | 作業を中断せず簡潔に答え、会話の本筋に影響させない |
| 「慎重に」「じっくり考えて」「大事なタスク」「複雑な問題」 | 深く考えてから回答する（ultrathink相当） |
| 「サクッと」「軽く」「簡単に教えて」 | 手短に答える（高速モード相当） |
| 「全体像を見せて」「計画してから」「大きなタスク」 | 実行前に計画を提示してから進める |
| 「戻して」「やり直し」「一個前に戻りたい」 | `/rewind` の案内をする |
| NotionのプロジェクトDBのプロパティが追加・変更されたと分かった時 | `fix_notion_data.yml` や関連ワークフローへの対応が必要か確認して提案する |

## 定義

定義が生まれたタイミングでここに追記すること。pushをトリガーにNotionの定義DBへ自動同期される。

### ルール
- 既存の定義の変更・削除にはユーザーの承認が必要
- 新規追加はClaudeが判断して追記可能
- 会話中に定義候補となる概念・判断・気づきが生まれたら、Claudeがその場で「これ定義にしませんか？」と提案する。ユーザーはYes/Noだけ答えればよい。Yesなら自動で追記してcommit/pushまで行う
- スコープが「共通」の定義は連携リポジトリに自動同期される。スコープがリポジトリ名の定義はそのリポジトリ内にとどまる

### 定義一覧

定義はNotionが管理元。セッション開始時および定義の話題が出た際に `python3 scripts/fetch_definitions.py` で取得する。
